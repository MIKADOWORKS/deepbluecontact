# 全エージェント共通ガイドライン

## 1. 行動原則

- オーナーの意図を最優先する
- **聖域ルール（CLAUDE.md 参照）**:
  1. ゲーム内通貨・時短課金は一切なし — 全てプレイ進行で解放
  2. 観察モードのボーナスは時間短縮のみ — 発見率・経験値には影響しない
  3. 教育コンテンツはゲーム進行に一切影響しない「添え物」
  4. 深海生物は実写写真を使用 — ピクセルアートにしない
  5. 同時派遣は1隻のみ
- 小さく・確実に（1タスク = 関連ファイル数本以内）
- 壊す前に聞く（破壊的変更は事前承認必須）
- 推測で動かない — 不明点はディレクターに確認する

## 2. コミュニケーションルール

- 結論ファースト（最初の1-2行で「何をした/何がわかった」）
- 長文禁止、簡潔に
- 問題は隠さない — エラー・課題は即報告
- 次のアクションを提案する
- `templates.md` の該当テンプレートに沿って報告する

## 3. コード品質基準（Godot / GDScript）

- GDScript の命名規則に従う（snake_case for variables/functions, PascalCase for classes/nodes）
- `class_name` を適切に使用する
- 型ヒント（type hints）を積極的に使う: `var speed: float = 10.0`
- シグナルは `signal_name` の snake_case で命名
- `@export` を使ってエディタから調整可能にする（ハードコード禁止）
- `@onready` を使ってノード参照を取得する
- `_ready()`, `_process()`, `_physics_process()` の使い分けを意識する
- リソース（.tres, .tscn）はパス管理を徹底する

## 4. ファイル操作ルール

| ロール | 読み取り可 | 書き込み可 |
|--------|-----------|-----------|
| ゲームプログラマー | 全ソースコード、リソース | 担当のシーン・スクリプト |
| UIエンジニア | 全ソースコード、リソース | UI関連シーン・スクリプト |
| QA | 全ファイル | バグ報告書のみ（コード修正はプログラマーへ依頼） |
| レベルデザイナー | ゲームデータ、バランス関連 | レベルデータ、バランスパラメータ |
| サウンドデザイナー | オーディオ関連ファイル | オーディオバス設定、サウンド関連スクリプト |
| リサーチャー | 全ファイル | なし（調査報告のみ） |
| マーケター | docs/、ゲーム情報 | docs/ 内マーケ資料 |
| コピーライター | docs/、ゲーム情報 | docs/ 内テキスト成果物 |
| コミュニティマネージャー | docs/、ゲーム情報 | docs/ 内運用資料 |

- **承認が必要な操作**: プロジェクト設定（project.godot）の変更、ディレクトリ構造の変更、外部アドオンの追加

## 5. やってはいけないこと（全ロール共通）

- ディレクター承認なしにプロジェクト設定を変更する
- 他ロールの担当ファイルを無断で書き換える
- コアゲームループの仕様を独断で変更する
- 未検証の外部アドオン・プラグインを追加する
- テストせずにコードをコミットする
- 仮データやデバッグコードを残したまま報告する
- 巨大なアセット（100MB超）をリポジトリに追加する

## 6. 技術的注意事項（Godot 4.x）

- Godot 4.x は 3.x と大幅にAPI が異なる — 3.x のコードをそのまま使わない
- `yield` は廃止 → `await` を使う
- `onready` → `@onready`、`export` → `@export`
- `KinematicBody2D` → `CharacterBody2D`
- `Tween` はノードではなく `create_tween()` で生成する
- `move_and_slide()` は引数なし（velocity はプロパティ）
- シグナル接続は `signal.connect(callable)` 形式を使う
- Autoload（シングルトン）は慎重に — 乱用しない
- Steam SDK 連携は GDExtension または GodotSteam プラグインを使用
